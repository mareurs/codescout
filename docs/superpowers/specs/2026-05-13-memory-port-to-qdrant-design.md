# L-01 — Port `memory.*` from sqlite-vec to Qdrant `memories` collection

**Status:** draft v2 — decisions proposed, awaiting review · **Opened:** 2026-05-13 · **Owner:** TBD

## Why

`src/tools/memory/mod.rs` is the last load-bearing consumer of
`src/embed/index.rs`. Until memory is on Qdrant, codescout cannot drop
`tantivy`, `sqlite-vec`, `fastembed`, or the `local-embed` feature — even
though they are no longer needed for code search.

Concretely, the blocker is the call surface:

```
src/tools/memory/mod.rs
  ↳ crate::embed::index::open_db(&root)
  ↳ crate::embed::index::ensure_vec_memories(conn)
  ↳ crate::embed::index::upsert_memory_by_title(...)
  ↳ crate::embed::index::ensure_memory_anchors(conn)
  ↳ crate::embed::index::delete_semantic_anchors(...)
  ↳ crate::embed::index::insert_semantic_anchor(...)
  ↳ crate::embed::index::search(conn, &embedding, top_n)
  ↳ crate::embed::index::get_file_hash(conn, path)
  ↳ crate::embed::index::delete_memory(conn, id)
```

Nine call sites in `memory/mod.rs` plus two in `src/prompts/builders.rs`. The
embedding source itself already routes through `codescout_embed::embed_one`
(any `Embedder`), so the work is purely on the **storage** side.

## Scope

| Item | Today | Target |
|---|---|---|
| Storage | sqlite-vec table `vec_memories` + side tables | Qdrant collection `memories` (per project) |
| Dense vectors | embedded via active `Embedder` | unchanged |
| Schema | `bucket, title, content, anchor_path, anchor_hash, created_at, updated_at` | same fields in Qdrant payload |
| Operations | open_db, ensure_*, upsert_by_title, search, delete | upsert / search / delete on `RetrievalClient` |
| Anchors | `semantic_anchors` table | second collection `memory_anchors` *or* embedded array on the memory point |
| Migration | n/a | one-shot tool reading sqlite-vec and bulk-importing into Qdrant |

## Open questions

- One collection per project, or one shared collection with `project_id` payload filter?
- Anchors as a separate collection, or embedded array on the memory point?
- Migration UX: opt-in CLI command, or automatic on first run?
- `index(action='build')` — re-route to `sync_project` semantics, or delete the tool?

## Answers (proposed)

### Q1 — Shared collection with `project_id` payload filter

**Decision:** single Qdrant collection `memories`, filter by `project_id`.

Mirrors how `code_chunks` already works in `src/retrieval/`. One schema, one
migration path, one place for indexes/backups. The isolation argument for
per-project collections doesn't justify diverging from the existing pattern —
codescout's per-project deletion is `points/delete` on a payload filter, not
collection drop, in either world.

### Q2 — Anchors embedded as a payload array on the memory point

**Decision:** each memory point carries `anchors: [{path: String, hash: String}]`
in its payload. No second collection.

Memories average ≤3 anchors. Embedded keeps memory + anchors atomic on a
single point — upsert is one Qdrant write instead of two. The cost is losing
the inverted "find all memories anchored to path X" query, which today is
served by a table scan anyway. Re-implement that as a `scroll(filter=anchors.path == X)`
on Qdrant when needed (rare).

### Q3 — Opt-in migration via `codescout migrate-memories`, with a smart hint

**Decision:** ship a one-shot subcommand. On first `memory(action='recall')`
after upgrade, if the Qdrant `memories` collection is empty for the active
project AND a legacy sqlite-vec store exists at `.codescout/projects/<id>/index.db`,
return `RecoverableError` with hint pointing at `codescout migrate-memories`.

Rationale: silent automatic migration on first MCP call risks slow startup
and unclear failure modes. Opt-in keeps the upgrade explicit, but the hint
makes discovery one error away — no manual reading of release notes.

The migration command iterates the sqlite-vec rows, re-uses the active
embedder to either copy or re-embed (configurable: copy is faster, re-embed
catches model changes), upserts into Qdrant, then renames the sqlite file
to `index.db.legacy` so it can be deleted later by hand.

### Q4 — Re-route `index(action='build'|'status')` to `sync_project` semantics

**Decision:** keep the tool, change the body.

Agents have system prompts that say "run `index(action='build')` to index a
project" — generated by `build_system_prompt_draft()`. Breaking that call
surface forces every onboarded project to refresh its prompt. Re-routing the
tool to call `RetrievalClient::sync_project` preserves the contract.

Response shape changes (no sqlite-vec row counts; report Qdrant point count
and last-synced timestamp), so the tool's `description()` and prompt
surfaces (`server_instructions.md`, `onboarding_prompt.md`,
`build_system_prompt_draft()`) update too — and onboarding version bumps,
per `CLAUDE.md § Prompt Surface Consistency`.

`scope='lib:<name>'` already routed elsewhere (see L-12 in the parent
tracker) — keep that wiring.

## Schema details

**Collection name:** `memories`
**Vector config:** dense only, dim = `CODESCOUT_MODEL_DIM`, distance `Cosine`.
No sparse vector — memories are short and the dense+rerank pipeline is enough.

**Payload schema:**

```jsonc
{
  "project_id": "code-explorer",           // shared-collection filter key
  "bucket": "code|system|preferences|unstructured",
  "title": "Parallel write safety (BUG-021)",
  "content": "NEVER dispatch parallel write tool calls...",
  "anchors": [                              // 0..N file anchors
    { "path": "src/tools/memory/mod.rs", "hash": "ab12…" }
  ],
  "created_at": "2026-03-03T12:00:00Z",
  "updated_at": "2026-05-13T05:00:00Z"
}
```

**Point ID:** UUID-v5 over `(project_id, bucket, title)` — gives idempotent
upsert without a separate "find by title" round-trip. Same trick code_chunks
uses for path-hashing.

**Required indexes:** payload field index on `project_id` (mandatory filter),
`bucket` (frequent filter), `anchors.path` (occasional reverse lookup).

## Trait + module sketch

```rust
// src/memory/semantic_store.rs
// Note: src/memory/mod.rs already has a MemoryStore for topic-based file
// memories (markdown in .codescout/memories/). The trait is named distinctly
// to avoid collision.
#[async_trait]
pub trait SemanticMemoryStore: Send + Sync {
    async fn upsert(&self, m: &SemanticMemory) -> Result<()>;
    async fn search(&self, project_id: &str, query: &[f32], top_n: usize,
                    bucket: Option<&str>) -> Result<Vec<SemanticMemory>>;
    async fn delete(&self, project_id: &str, id: Uuid) -> Result<()>;
    async fn list(&self, project_id: &str) -> Result<Vec<SemanticMemory>>;
    async fn by_anchor(&self, project_id: &str, path: &str) -> Result<Vec<SemanticMemory>>;
}

// src/memory/semantic_store_qdrant.rs — production impl using RetrievalClient
// src/memory/semantic_store_sqlite.rs — legacy impl, feature-gated `legacy-vec`
```

`src/tools/memory/mod.rs` (the tool implementation — 9 sites) switches from
`crate::embed::index::*` to `ctx.agent.semantic_memory_store()` which returns
the configured impl. Default impl is Qdrant; users on `--features legacy-vec`
can keep sqlite while migrating.

## Implementation order

1. **This doc + review.** Lock decisions before code.
2. **Add `memories` collection bootstrap** to `QdrantWrap::ensure_collection`.
   Add `RetrievalClient::memory_upsert`, `memory_search`, `memory_delete`,
   `memory_list`, `memory_by_anchor`.
3. **Define `MemoryStore` trait + Qdrant impl.** Tests against a Qdrant
   testcontainer. ~300 LOC + tests.
4. **Switch `memory/mod.rs` (9 sites) + `prompts/builders.rs` (2 sites)** to
   the trait. Run the full memory test suite; expect 0 functional changes.
5. **`migrate-memories` subcommand.** Read sqlite-vec, embed each row,
   bulk-upsert to Qdrant, rename old db. Idempotent (skip already-imported).
6. **Re-route `index` tool to `sync_project`.** Update three prompt surfaces.
   Bump `ONBOARDING_VERSION` in `src/tools/onboarding.rs`.
7. **Smart-hint wiring**: detect legacy db on `memory(recall)`, return
   `RecoverableError` with migrate command.
8. **Delete legacy**: `src/embed/{index,bm25,fusion,local,drift}.rs`,
   `src/embed/schema.rs` (if not used by memory store), drop `tantivy`,
   `sqlite-vec`, `fastembed` from `Cargo.toml`. Flip `local-embed` default
   off (or delete the feature). One-shot mechanical PR.

Each step lands as its own PR. Steps 1-3 are independent of any user
upgrade pain. Step 4 must not regress memory semantics (covered by tests).
Step 8 is the binary diet that motivated this work.

## Risks

- **Embedder availability during migration.** If migration runs without the
  retrieval stack up, it errors. Mitigation: `migrate-memories` checks
  `CODESCOUT_EMBEDDER_URL` health first.
- **Existing semantic anchors with empty hash.** Some legacy rows have
  `anchor_hash = NULL`. Migration must tolerate and re-hash from file at
  current commit.
- **Project rename**: when `[project].name` changes in `project.toml`, the
  `project_id` filter key changes. Migration tool must accept
  `--rename old=new` to relabel points without re-embedding.
- **Memory tool tests live in `src/tools/memory/tests.rs`.** They likely
  spin up a real sqlite-vec db; need re-targeting at a Qdrant testcontainer.
  Audit before step 3.
